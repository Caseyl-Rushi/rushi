# 如是 · 游戏设计文档
## 第八章：技术方案

> **核心原则**：每一阶段只用最必要的技术。不提前优化，不盲目追新。技术的选择服务于"你能一个人做到"，而不是"理想状态下最佳方案"。

### 二、分阶段技术栈

#### 阶段 3：技术原型（单机版）

**目标**：一个人、一浏览器、一个 HTML 文件。能跑就行。

| 层级 | 技术选择 | 理由 |
|------|---------|------|
| 渲染 | HTML5 Canvas 2D | 浏览器原生支持，零依赖，学习资源极多。斜 45° 等距瓦片用 Canvas 完全够用 |
| UI 层 | DOM（HTML + CSS） | 日程本、App、公告板用网页元素实现，比 Canvas 画 UI 容易十倍 |
| 游戏逻辑 | 原生 JavaScript | 不引入框架。原型阶段代码量小，框架的学习成本比收益高 |
| 数据存储 | localStorage | 浏览器自带，存 JSON。单机不需要数据库 |
| 文件结构 | 1 个 HTML + 1 个 JS + 1 个 CSS | 不是偷懒——是保持简单。等你需要拆文件的时候自然会拆 |
| 开发工具 | VS Code + Cursor/Claude | VS Code 免费。AI 辅助大幅降低学习曲线 |

**不用的东西，和为什么不用的理由**：

| 不用什么 | 为什么 |
|---------|--------|
| React / Vue | 原型不需要组件化框架。DOM 操作用原生 JS 直接写，更直观 |
| PixiJS / Phaser | Canvas 原生就够用。等需要性能优化或复杂动画时再引入 |
| TypeScript | 先学好 JS。类型系统是好东西，但初学阶段增加认知负担 |
| Webpack / Vite | 一个文件不需要打包工具。浏览器直接打开 |
| Node.js 后端 | 单机版没有多人，不需要服务器 |

---

#### 阶段 5：Alpha 版本（多人版）

**目标**：10-30 人同时在线。最小客户端 + 简单后端。

| 层级 | 技术选择 | 理由 |
|------|---------|------|
| 渲染 | HTML5 Canvas + PixiJS（可选） | 如果 Canvas 原生性能不够，PixiJS 是轻量替代。API 友好 |
| UI 层 | DOM | 不变 |
| 前端逻辑 | 原生 JS 或轻量框架（如 Alpine.js） | 保持简单。React 也可以但非必须 |
| 后端 | Node.js + Express | 和前端同语言（JS），学习成本最低。Express 最简单 |
| 实时通信 | WebSocket（Socket.IO） | 玩家在街上的位置、实时投票数据、NPC 行为同步 |
| 数据库 | SQLite（开发）→ Supabase / PostgreSQL（部署） | SQLite 零配置，适合本地开发。Supabase 有免费额度，托管 PostgreSQL + 实时功能 |
| 用户认证 | 简单 Token 认证或 Supabase Auth | 不自己做密码存储——用成熟方案 |
| 部署 | 单台 VPS 或 Supabase + Vercel/Netlify | Alpha 阶段不需要 K8s / 微服务。单机部署完全够 |

**Alpha 不用的东西**：

| 不用什么 | 为什么 |
|---------|--------|
| Docker / K8s | 30 人在线不需要容器编排 |
| Redis | SQLite/PostgreSQL 的查询缓存够用 |
| CDN | 静态资源直接从服务器发。人数少不需要 |
| 微服务 | 所有逻辑一个 Node 进程跑。拆服务是百人以上才需要考虑的事 |

### 三、关键技术决策详解

#### 为什么是全栈 JavaScript

前端 Canvas + JS，后端 Node.js。一门语言贯穿。你只需要学 JavaScript——不是"学 JS + 再学 Python + 再学 C#"。这是你零基础起步的最大加速器。

#### 为什么存储从 localStorage 起步

- 零配置。浏览器自带
- 数据模型用 JSON，和 JS 对象天然一致
- 存读性能对单机完全够用
- 缺点是数据在浏览器里，清缓存就没了。但原型阶段这是可接受的

#### 为什么 Alpha 选 Supabase 而不是自己搭数据库

- Supabase 免费额度：500MB 数据库 + 50,000 月活用户。Alpha 阶段完全够
- 自带 REST API 和实时订阅——你不需要自己写 API 层
- 自带认证系统——你不需要自己处理密码加密
- 自带行级安全——你可以定义"玩家只能修改自己的数据"
- 把"运维数据库"这件事外包了，你专注写游戏逻辑

#### 为什么不用游戏引擎（Unity / Unreal / Godot）

| 方面 | 游戏引擎 | Web 技术 |
|------|---------|---------|
| 学习曲线 | 陡峭。每个引擎有自己的编辑器、脚本语言、渲染管线 | JS 是世界上最多人学的语言。HTML/CSS 有最多的学习资源 |
| 分发 | 需要安装包 / 平台审核 | 一个链接就能玩 |
| 跨平台 | 需要各平台编译 | 有浏览器就能跑 |
| AI 辅助 | AI 对游戏引擎的支持参差不齐 | AI 对 Web 技术的支持最成熟 |
| 适合场景 | 3D、高性能、复杂物理 | 2D、信息密集型、社交型 |

如是是 2D、信息密集型、社交型产品。不需要 3D 引擎。用 Web 技术是正确的。

### 四、关键模块的技术实现思路

#### 斜 45° 等距渲染

**坐标转换**：
```
// 屏幕坐标 → 等距地图坐标
function screenToIso(screenX, screenY) {
  // 标准等距转换：旋转 45°，压缩 Y 轴至 50%
  let mapX = (screenX / tileWidth + screenY / tileHeight) / 2;
  let mapY = (screenY / tileHeight - screenX / tileWidth) / 2;
  return { x: Math.floor(mapX), y: Math.floor(mapY) };
}

// 等距地图坐标 → 屏幕坐标
function isoToScreen(mapX, mapY) {
  let screenX = (mapX - mapY) * tileWidth / 2;
  let screenY = (mapX + mapY) * tileHeight / 2;
  return { x: screenX, y: screenY };
}
```

**渲染顺序**：按 Y 坐标从小到大渲染（"先画上面的，再画下面的"），自动处理遮挡。

**有机布局实现**：每个格位存储 (mapX, mapY) 坐标 + offsetX/offsetY 微调值。格位不要求严格对齐。

#### NPC 行为调度

```
// 简单状态机
class NPC {
  constructor(name, home, workplace) {
    this.state = 'idle';  // idle | working | walking | resting | talking
    this.schedule = [
      { time: 6,  state: 'working',  location: workplace },
      { time: 12, state: 'walking',  location: park },
      { time: 13, state: 'resting',  location: park },
      { time: 14, state: 'walking',  location: workplace },
      { time: 18, state: 'walking',  location: home },
      { time: 22, state: 'sleeping', location: home },
    ];
  }
  
  update(currentTime) {
    // 根据时间表切换状态和位置
    // 在 walking 状态下，逐帧沿路径移动
  }
}
```

**AI 对话生成**（Alpha 中期加入）：
- 调用 AI API（如 Claude）生成 NPC 对话
- 输入：NPC 人设 + 当天街区动态 + 与玩家的互动历史
- 输出限制：2-3 句话。不追问，不命令，不和玩家尬聊
- 缓存机制：同一情境下的对话缓存一定时间，减少 API 调用

#### 多人同步

**Socket.IO 事件流**：

| 事件 | 发送方 | 接收方 | 数据 |
|------|--------|--------|------|
| player:move | 客户端 | 服务器 → 同一街区其他客户端 | { playerId, mapX, mapY } |
| proposal:submit | 客户端 | 服务器 → 全部 | { proposalId, title, author } |
| vote:cast | 客户端 | 服务器 → 全部 | { proposalId, optionId }（不暴露谁投了什么） |
| npc:dialogue | 服务器 | 附近客户端 | { npcId, text, targetPlayerId } |
| world:timeskip | 服务器 | 全部 | { daysSkipped, reason } |
| bulletin:update | 服务器 | 全部 | { newItems[] } |

**权威服务器**：所有游戏逻辑计算在服务器端。客户端只负责渲染和发送输入。防止作弊。

#### AI 地面文明引擎

地面世界不是另一个等距地图，它是一个**AI 驱动的事件引擎**。

**架构**：
```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  AI 事件引擎  │ ──→ │  世界状态数据库 │ ──→ │  天空之城客户端 │
│              │     │              │     │              │
│ 生成事件      │     │ 存储星球状态   │     │ 悬赏/议题展示  │
│ 推演后果      │     │ 记录历史事件链 │     │ 玩家投票输入   │
│ 输出叙事      │     │ 繁荣度等指标   │     │ 结果反馈显示   │
└──────────────┘     └──────────────┘     └──────────────┘
       ↑                                        │
       └────────── 投票结果回传 ─────────────────┘
```

**事件生成流程**（每次触发周期约 1-3 天）：
1. AI 基于当前世界状态 + 文明阶段 + 随机种子，生成 1-3 个待发生事件
2. 事件被转化为"地面悬赏"，出现在天空之城的议事厅
3. 玩家投票/提案
4. 投票结果 + 上下文 → 回传 AI → AI 推演后果
5. 后果以叙事文本 + 数据变化的形式显示

**AI 模型选择**：
- 原型阶段：Claude API / GPT API——最成熟的长文本推理
- 正式阶段：考虑国产大模型（降低成本 + 符合国内合规要求）

**成本预估**：
- 每次事件链（生成+推演+叙事）：约 500-3000 token
- 按每周 20 次事件计算，月成本约 ¥10-50
- 地面世界不需要实时渲染，不需要多人同步——主要成本是 AI API 调用

**关键设计约束**：
- AI 不对玩家方案做价值判断——只推演后果
- 事件链必须有因果连续性——上一次投票的结果必须在后续事件中体现
- 叙事文本不喊口号、不贴标签——让玩家自己得出结论

### 六、部署方案

#### 阶段 3（原型）：零部署

HTML 文件存本地，浏览器打开。如果你要给别人看原型，用以下任一方式：

- 把 HTML 文件发过去（对方浏览器打开）
- 用 GitHub Pages（免费）——把文件 push 到 GitHub，自动获得一个网址
- 用 Netlify Drop——把文件夹拖进网页，获得一个临时网址

#### 阶段 5（Alpha）：基础部署

```
方案 A（最简）：
  ┌─────────────┐     ┌─────────────────┐
  │  Vercel       │ ←── │  GitHub（前端代码）│
  │  （前端托管）   │     └─────────────────┘
  └──────┬──────┘
         │
  ┌──────┴──────┐
  │  Supabase     │  （数据库 + 认证 + 实时）
  └─────────────┘

方案 B（更多控制）：
  ┌─────────────┐
  │  VPS 单机     │  运行 Node.js + PostgreSQL
  │  （如阿里云）  │  30 人以内单机完全够用
  └─────────────┘
```

推荐方案 A。零运维。免费额度覆盖 Alpha。
