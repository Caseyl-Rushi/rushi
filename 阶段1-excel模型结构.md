# 如是 · 阶段 1 纸面验证 — Excel 模型结构

> 用途：在代码写一行之前，用表格跑一遍经济系统是否自洽。
> 工具：Excel / WPS 表格
> 规模：3 个街区 × 10 个居民/街区 = 30 人，模拟 4 周（28 天）

---

## 一、模型总览

```
┌─────────────────────────────────────────────┐
│  Sheet 1：全局参数                           │  ← 所有数值的"旋钮"
├─────────────────────────────────────────────┤
│  Sheet 2：居民档案（30 行）                   │  ← 每个居民的属性
├─────────────────────────────────────────────┤
│  Sheet 3：每日流水（28 列 × 30 行）           │  ← 每人每天的收支和参与
├─────────────────────────────────────────────┤
│  Sheet 4：街区汇总（4 周 × 3 街区）          │  ← 每个街区的每日/每周数据
├─────────────────────────────────────────────┤
│  Sheet 5：系统健康度                         │  ← 四个验证指标的结论
└─────────────────────────────────────────────┘
```

---

## 二、Sheet 1：全局参数（所有数值从这里读取）

建一个表，把以下参数填进去。**后续调参只改这里，其他 sheet 自动联动。**

### A. 时间参数

| 参数 | 初设值 | 说明 |
|------|--------|------|
| 模拟天数 | 28 | 4 周 |
| 每周悬赏数 | 1 | 每个街区每周 1 个悬赏 |
| 悬赏提案期（天） | 3 | |
| 悬赏投票期（天） | 2 | |

### B. Token 参数

| 参数 | 初设值 | 来源 |
|------|--------|------|
| 日配额（新手） | 100 | 第三章 |
| 周补充 | 300 | 第三章 |
| 投票消耗（每次） | 10 | 第三章 |
| 发言消耗（每次） | 5 | 第三章 |
| 提案消耗（次） | 50 | 第三章 |
| 自提方案消耗 | 30 | 第四章 |
| 认可他人消耗 | 1 → 返还 2 | 第三章 |
| 不累积 | 是 | 当日/当周未用完清零 |

### C. 贡献值参数

| 参数 | 初设值 | 来源 |
|------|--------|------|
| 悬赏采纳奖励 | 200 | 第四章 |
| 自提方案通过奖励 | 100 | 第四章 |
| 投票权等级 0-49 | 1 票 | 第三章 |
| 投票权等级 50-199 | 3 票 | 第三章 |
| 投票权等级 200-499 | 6 票 | 第三章 |
| 投票权等级 500+ | 10 票 | 第三章 |

### D. 通用货币参数

| 参数 | 初设值 | 来源 |
|------|--------|------|
| 新手初始货币 | 500 | 第三章 |
| 每日基础分配 | 50 | 第三章 |
| 面包价格 | 5-15 | 待定 |
| 杂货铺基础商品价格 | 10-30 | 待定 |
| 交易税率 | 5% | 第三章 |

### E. 经营产出参数

| 职业 | 日产出范围（通用货币） | 说明 |
|------|---------------------|------|
| 面包店 | 20-80 | 取决于客流量（随机波动 ±30%）|
| 书店 | 15-60 | 客流少但单价高 |
| 杂货铺 | 25-90 | 高频低利 |
| 花店 | 10-40 | 低频高利 |
| 茶馆 | 30-120 | 晚间营业 |
| 修理摊 | 18-70 | 不稳定 |
| 诊所 | 0（NPC） | 不产出货币，产出社会价值 |
| 无职业（纯居民） | 0 + 基础分配 50 | 只靠系统分配 |

> **注意**：经营产出是"毛收入"。实际净收入还需扣除"进货成本"（约占毛收入的 30%）。你可以先简化为净收入 = 毛收入 × 0.7。

## 三、Sheet 2：居民档案（30 行）

每一行是一个居民。列如下：

| 列 | 列名 | 内容 | 公式/填写 |
|----|------|------|----------|
| A | ID | P01 - P30 | 手动编号 |
| B | 所在街区 | 第 7 / 第 3 / 第 5 | 手动分配，每街区约 10 人 |
| C | 类型 | 玩家 / NPC | 前 5 个是 NPC（阿琳、林教授、陈医生、杂货铺老板、旅行商人），其余 25 个是玩家 |
| D | 职业 | 见上表 E | 手动分配。NPC 有固定职业。玩家随机或按分布分配 |
| E | 初始贡献值 | 0（新手） | 全部从 0 开始 |
| F | 初始货币 | 500 | 全部相同 |
| G | 活跃类型 | 积极型 / 中间型 / 佛系 | **关键变量——决定行为模式**（见下文） |
| H | 活跃概率(%) | 80 / 50 / 20 | 对应活跃类型 |

### 活跃类型定义

| 类型 | 含义 | 占比建议 |
|------|------|---------|
| 积极型 | 几乎每天登录，参与悬赏、投票、发言 | 约 20%（6 人） |
| 中间型 | 经常登录，偶尔参与公共事务 | 约 50%（15 人） |
| 佛系 | 偶尔登录，主要经营自己的店，很少投票 | 约 30%（9 人） |

每天模拟时，根据活跃概率随机决定该玩家今天是否"上线"。不上线 = 不产生任何操作和数据变化（除了基础货币分配照常到账）。

## 四、Sheet 3：每日流水（核心表）

这是最大的表。结构：

**行**：30 个居民（每人一行）
**列**：28 天（每天一列组），每组包含以下子列：

### 每天的子列（以 Day 1 为例）：

| 子列 | 内容 | 计算方式 |
|------|------|---------|
| D1-上线? | 今天是否上线 | RAND() < 活跃概率 → 是/否 |
| D1-Token余额(始) | 今日开始时的 Token | = 昨日剩余（Day1 = 日配额 100） |
| D1-经营收入 | 经营产出 | 上线? → 职业日产出 × (0.7~1.0 随机) ; 不上线 → 0 |
| D1-基础分配 | 系统赠送 | 50（无论是否上线） |
| D1-交易支出 | 买东西花了多少 | 上线? → 随机 0~30 ; 不上线 → 0 |
| D1-投票? | 是否参与了投票 | 上线? 且 有进行中悬赏? 且 RAND()<某概率 → 是 |
| D1-投票消耗 | 投票花掉的 Token | 投票? → 10 × 投票票数 ; 否则 → 0 |
| D1-发言? | 是否在议事厅发言 | 上线? 且 中间型/积极型且 RAND()<0.3 → 是 |
| D1-发言消耗 | 发言花掉的 Token | 发言? → 5 ; 否则 → 0 |
| D1-提方案? | 是否提交了方案 | 上线? 且 积极型 且 本周有悬赏 且 RAND()<0.2 → 是 |
| D1-提方案消耗 | 提案花掉的 Token | 提方案? → 50 ; 否则 → 0 |
| D1-认可他人? | 是否认可了别人 | 上线? 且 RAND()<0.5 → 是 |
| D1-认可净消耗 | 认可花掉/赚到的 Token | 认可? → -1+2=+1 ; 否则 → 0 |
| D1-Token总消耗 | 今日 Token 总消耗 | = 投票+发言+提方案+认可 的代数和 |
| D1-Token余额(末) | 今日结束时的 Token | = 余额(始) - 总消耗 + 认可净赚 |
| D1-货币余额(末) | 今日结束时的货币 | = 昨日货币 + 经营收入 + 基础分配 - 交易支出 |
| D1-贡献值变化 | 今日贡献值变动 | = 提案被采纳? → +200 : 0 （第一周大概率没人被采纳） |
| D1-贡献值累计 | 累计贡献值 | = 昨日累计 + 变化 |

### 简化版（如果你觉得上面太复杂）

如果完整版公式太多，先用这个**极简版**跑通：

| 子列 | 内容 | 极简计算 |
|------|------|---------|
| 上线? | 是/否 | 随机 |
| Token消耗 | 今日总消耗 | 上线? → 随机 20~80 ; 不上线 → 0 |
| Token余额 | 结余 | = 昨日余额 - 消耗（<0 则记为 0，不能负） |
| 货币变化 | 今日收支 | = 经营收入(如上线) + 50 - 支出(如上线) |
| 货币余额 | 结余 | = 昨日 + 变化 |
| 贡献值变化 | 今日获得 | 上线 && 积极 && 随机小概率 → +200 ; 否则 0 |
| 贡献值累计 | 总计 | = 昨日 + 变化 |

## 五、Sheet 4：街区汇总

每个街区每天/每周的数据汇总。

### 每日汇总（以第 7 街区 Day 1 为例）：

| 指标 | 计算方式 |
|------|---------|
| 在线人数 | 该街区今天上线? = "是"的人数总和 |
| 总 Token 消耗 | 该街区所有居民 D1-Token总消耗 之和 |
| 总货币流通量 | 该街区所有居民 D1-交易支出 之和（= 市集的交易额） |
| 总贡献值产生 | 该街区所有居民 D1-贡献值变化 之和 |
| 投票参与率 | 投票? = "是" 的人数 / 在线人数 |
| 提案提交数 | 提方案? = "是" 的人数 |

### 每周汇总（4 个周汇总行）：

| 指标 | 计算方式 |
|------|---------|
| 周平均在线率 | 7 天在线人数均值 / 街区总人口 |
| 周总 Token 消耗 | 7 天消耗之和 |
| 周总货币增量 | 7 天（经营收入+基础分配-支出）之和 |
| 周新增贡献值总量 | 7 天贡献值变化之和 |
| 周悬赏参与率 | 有提案提交的天数 / 7 |
| 周投票参与率 | 有投票行为的天数 / 7 |

## 六、Sheet 5：系统健康度（最终结论）

跑完 28 天后，这张表自动给出四个验证结果：

### 验证 A：Token 是否够用

| 检查项 | 通过标准 | 不通过的表现 |
|--------|---------|-------------|
| 新手日均剩余比例 | ≥ 30% | 大部分新手每天 Token 归零 |
| 无人因 Token 不足完全无法参与 | 佛系玩家也能偶尔投一票 | 佛系玩家连续 7 天 Token 消耗 = 0（说明门槛太高）|

### 验证 B：贡献值集中度

| 检查项 | 通过标准 | 不通过的表现 |
|--------|---------|-------------|
| TOP3 占总贡献值比例 | < 50% | 前 3 名占了超过一半的贡献值 |
| 佛系玩家 4 周后贡献值 > 0 | 至少有几个佛系玩家因为偶然参与获得了贡献值 | 佛系玩家全部 = 0（说明只有积极型能获得贡献值）|

### 验证 C：经济通胀

| 检查项 | 通过标准 | 不通过的表现 |
|--------|---------|-------------|
| 28 天后全区货币总量 / 初始总量 | < 3 倍 | 超过 3 倍（严重通胀） |
| 有人货币归零 | < 10% | 超过 3 个人货币 ≤ 0 |

### 验证 D：是否存在"死号"

| 检查项 | 通过标准 | 不通过的表现 |
|--------|---------|-------------|
| 所有人 4 周后货币 > 0 | 100% | 有人货币归零且无法恢复 |
| 所有人至少参与过一次投票/提案 | > 80% | 超过 20% 的人 4 周内零参与 |

## 七、你现在的操作步骤

1. 打开 Excel / WPS
2. 新建一个文件，命名为 `如是-经济模拟.xlsx`
3. 创建 5 个 Sheet，分别命名为：`全局参数` `居民档案` `每日流水` `街区汇总` `系统健康度`
4. 按照 Sheet 1 的参数表，先把所有初设值填进去
5. 按照 Sheet 2 的格式，创建 30 行居民档案（前 5 个是 NPC，后面 25 个是玩家）
6. 先做 **极简版** 的每日流水（不用完整版）
7. 用 Excel 的 RAND() 函数模拟随机性
8. 跑完 28 天，看 Sheet 5 的四个验证结果
9. 如果不通过 → 回到 Sheet 1 调参数 → 重跑

---

## 八、需要你手动决定的几个事

在开始之前，你有三个选择要拍板：

1. **30 个居民里，NPC 和玩家的比例？** 我建议 5 NPC + 25 玩家。你觉得呢？
2. **经营产出要不要加随机波动？** 我建议加（±30%），否则数据太平滑，看不出真实情况
3. **先跑极简版还是完整版？** 我建议先极简版跑通，再加复杂度

这三个决定了就开干。
